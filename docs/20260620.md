## 1. SaveOrderBind SQL 占位符冲突（绑单 100% 失败）

### 现象

- gb12「Direborn vs Mentality Monster」两笔套利单（OB + RAY）在 UI 上**未关联同一 LinkID**，各自显示为独立占位 link。
- 客户端 `finalizeArbBet` 调用了 `saveOrderBind`，但 RDS 里 `link` 仍为 `create_at - 1` 占位值。
- 生产 error 日志可见：
  ```
  [rds] updateOrderBind: column "link" is of type bigint but expression is of type uuid
  ```

### 根因

`changmen/server/db/rds/orders_store.js` 中 `updateOrderBind` 的 UPDATE 语句：

```sql
UPDATE orders SET link = $1 WHERE user_id = $1 AND ...
```

PostgreSQL 占位符**全局编号**，两个 `$1` 实际绑定同一参数；`user_id`（uuid）被误写入 `link`（bigint）列，UPDATE 失败。

API 层仍可能返回 `ok: true`（旧逻辑未把绑单失败传回客户端）。

### 修复

- UPDATE 的 WHERE 子句占位符整体偏移：`offsetSqlPlaceholders(where, 1)`，参数顺序 `[linkVal, ...params]`。
- `account_service.js`：绑单失败时返回 `ok: false`。
- 单元测试：`orders_store.test.mjs`。

### 提交

- `c73f3d6` — fix(changmen): repair SaveOrderBind SQL and admin order UX

### 验证

- 新套利单成单后，两笔 `link` 应为同一套利 LinkID（≥ 1e12）。
- 手动：`node scripts/lookup-order-logs.mjs --user gb12 --link <linkId>`
- 历史单：`node scripts/auto-rebind-arb-orders.mjs --user gb12 --days 5 --execute`

## 3. 管理员 Telegram 未收到订单通知（多因叠加）

### 现象

- 管理员（chat `8504789346`）在 gb12 Direborn 成单时未收到通知。
- gb12 使用者（chat `7957197543`）也可能未收到浏览器「下单提醒」。

### 根因（按优先级）

| # | 原因 | 说明 |
|---|------|------|
| A | **upsert 失败**（见 bug #2） | INSERT hook 未触发，admin 通知不发 |
| B | **VPS 缺 `TELEGRAM_ADMIN_CHAT_ID`** | `.env` 仅有 `TELEGRAM_BOT_TOKEN`；`isAdminNotifyEnabled()` 为 false |
| C | **SaveOrderBind 失败**（见 bug #1） | bound hook 不触发；**不影响** INSERT 通知，但绑单后不补发 |
| D | **通道混淆** | 管理员通知走 `TELEGRAM_ADMIN_CHAT_ID`；gb12 `Message.telegramId` 是另一套浏览器「下单提醒」 |
| E | **成功无日志** | `[admin-tools:order]` **仅打失败**；日志里 0 条 ≠ 发送成功 |

### 修复

1. 修复 upsert + SaveOrderBind（#1、#2）。
2. 扩展 `sync-telegram-env.mjs` / `sync-telegram-env-remote.sh`：同步 **token + admin chat id** 到 VPS。
3. 新增 `post-deploy-check.mjs`，deploy 后自动验证 upsert + Telegram。

### 提交

- `1c57baa` — post-deploy check
- `6411420` — sync TELEGRAM_ADMIN_CHAT_ID to VPS

### 验证

```bash
# 本机同步 Telegram 环境到 VPS
cd changmen && node scripts/sync-telegram-env.mjs

# VPS 完整自检
ssh -i ~/.ssh/id_ed25519_gamebet root@47.82.100.166 \
  "cd /root/gamebet/changmen/server/backend && node scripts/post-deploy-check.mjs"
```

预期：`== PASS (7/7) ==`，管理员 chat 收到 **【管理员·Deploy自检】**。

### 说明：bound hook 与 `link = create_at - 1`

未绑单入库占位 `link = create_at - 1`（`backendBindLinkFromCreateAt`）**不会**触发 `ordersBoundHook`。  
管理员「新订单」依赖 **INSERT hook**；绑单成功后的 bound 通知仅对 hash / `link=create_at` / 入库时间占位生效。  
事后手动补绑**不会**补发「下单提醒」或已过期的 INSERT 通知。

## 4. post-deploy-check 在 VPS 上误报「无 RDS 连接池」

### 现象

deploy 末尾自检失败：

```
FAIL database — 无 DATABASE_URL / RDS 连接池
```

同一 VPS 上 `gamebet-web` 正常运行且能连 RDS。

### 根因

VPS `.env` 使用 `DATABASE_URL_INTERNAL` / `DATABASE_URL_PUBLIC`（无单独 `DATABASE_URL`）。  
`start-db.mjs` 会 `await initDatabaseUrl()` 探测内网/外网；自检脚本只调了 `loadChangmenEnv()` + `getPgPool()`，**未解析**连接串，`getPgPool()` 返回 `null`。

### 修复

`post-deploy-check.mjs` 在 `getPgPool()` 前增加 `await initDatabaseUrl()`。

### 提交

- `3534b59` — fix(changmen): call initDatabaseUrl in post-deploy-check

### 验证

deploy 日志应出现：

```
[db] RDS 使用内网 DATABASE_URL_INTERNAL (auto)
[db] RDS 连接池已就绪 (post-deploy-check)
OK  database
```
