# 安全与边界说明（当前风险清单）

本文档用于在实现“多用户 + 服务器部署”时，明确哪些点需要收敛，避免把调试便利变成生产漏洞。

## 1) 信任边界（最重要）

- **浏览器（不可信）**：用户可篡改请求、伪造参数、重放 token、构造异常 header。
- **后端（可信执行环境）**：应统一做鉴权、审计、白名单、限流、脱敏。
- **第三方场馆/服务（不可信外部）**：响应可能异常/恶意；需超时、重试与错误隔离。

## 2) 通用 HTTP Relay（SSRF 风险）

### 背景

项目中存在“后端代发 HTTP”的能力（浏览器带 `x-proxy-url`，后端去请求真实目标再返回）。

这类能力若不限制，会变成标准的 SSRF 漏洞入口（可探测内网、打云元数据、绕过防火墙）。

### 现象（需重点审计）

- 请求头：`x-proxy-url` 指向真实目标 URL（或 path）
- 可选 `x-proxy` 指向 socks 代理

### 生产必须做的收敛（建议按优先级）

- **P0：目标 URL 白名单**
  - 仅允许已知场馆域名集合（OB/RAY/…）
  - 禁止 `127.0.0.1`、`localhost`、`169.254.169.254`、内网网段、file:// 等
- **P0：路径前缀限制**
  - 对每个平台只允许特定 API 前缀（例如 OB 仅 `/game/`）
- **P0：用户鉴权 + 审计日志**
  - 每次 relay 都记录：userId / accountId / provider / target host / path / status / latency
- **P1：限流**
  - 按用户、按账号、按 provider 限速
- **P1：响应脱敏**
  - relay 返回前做关键字段脱敏（若响应包含凭证/个人信息）

## 3) 凭证与敏感信息

### 风险点

- 浏览器存储（localStorage/kv）天然可被用户读取/导出
- 日志/报错中可能泄露 token、代理用户名密码、cookie 等

### 建议

- **token 最小化可见范围**
  - 能放后端就放后端；前端只持有 session token
- **日志脱敏规范**
  - token、authorization、cookie、proxy url 中的 user/pass 一律打码

## 4) Chrome 扩展依赖（环境差异风险）

### 建议

- 扩展只承载“必须复用浏览器登录态/跨域 cookie”的能力；其余尽量服务端化
- UI 明确标注“缺扩展会降级哪些功能”

## 5) 安全验收清单（用于上线前 gate）

- [ ] relay 接口对 `x-proxy-url` 做域名白名单与内网禁止
- [ ] relay 接口对 path 做前缀白名单
- [ ] relay 接口要求登录态/鉴权（不可匿名）
- [ ] relay 全链路审计日志具备（含 userId/accountId）
- [ ] 日志中不落明文 token / 代理密码
- [ ] 前端构建产物不包含 `localhost/127.0.0.1` 的生产默认值
